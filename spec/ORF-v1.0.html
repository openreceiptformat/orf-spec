Introduction

The Open Receipt Format (ORF) is a standardized, JSON-based format for digital receipts. It enables interoperability between merchants, point-of-sale systems, and consumer applications while respecting user privacy.

Implementation Guidelines

For POS Systems

Generating ORF receipts:

  1. 1 After transaction completion, serialize transaction data to ORF JSON
  2. 2 Assign unique receipt_id (UUID recommended)
  3. 3 Include all available data, but only required fields are mandatory
  4. 4 Make receipt available via:
  • â€¢ NFC tag (NDEF format)
  • â€¢ QR code (URL pointing to receipt endpoint)
  • â€¢ API endpoint for programmatic access
  • â€¢ Email/SMS with embedded JSON or link

Storage recommendations:

  • â€¢ Retain receipts for minimum 90 days
  • â€¢ Longer retention for high-value items
  • â€¢ Comply with local data retention laws

For Consumer Applications

Parsing ORF receipts:

  1. 1 Validate orf_version is supported (currently only "1.0")
  2. 2 Check all required fields are present
  3. 3 Gracefully handle unknown fields (forward compatibility)
  4. 4 Store locally by default (privacy preservation)
  5. 5 Validate currency codes, country codes against standards

Storage recommendations:

  • â€¢ Store receipts in local database or file system
  • â€¢ Use receipt_id as primary key
  • â€¢ Index by timestamp, merchant, total for quick retrieval
  • â€¢ Implement search across items, merchants, categories

Versioning

ORF uses semantic versioning:

  • Major version (1.x): Breaking changes, old parsers won't work
  • Minor version (x.0): New optional fields, backward compatible
